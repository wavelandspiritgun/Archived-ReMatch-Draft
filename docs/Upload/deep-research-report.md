# Research-informed benchmark  
Drawing on best practices and examples for nonprofit and research sites, a high-quality design would **immediately communicate mission and credibility**【48†L21-L24】【36†L26-L35】. Leading nonprofit sites (e.g. large NGOs) use clean, modern designs, prominent calls-to-action (like a clear *Donate* or *Submit Report* button), and strong imagery or logos to establish trust【45†L33-L40】【45†L59-L62】. They also optimize for accessibility (only ~22% of nonprofits do today, so doing so stands out【45†L64-L68】) and provide clear navigation and purpose. Branding should be consistent and distinctive: “consistent visual branding… assures visitors that the content… is yours”【48†L72-L76】. In contrast, bland or generic layouts and copy (common with AI-generated content) will erode confidence: “bland, generic content… reads like it was generated from a quick prompt”【38†L177-L184】. 

Form design must follow solid UX rules. Top guidelines include *keeping forms as short as possible*【50†L131-L139】, *grouping related fields with labels adjacent to inputs*【52†L1-L4】, and *using a single-column layout*【53†L1-L4】 to maintain a smooth vertical flow. Forms that appear visually attractive and address user concerns (for example, reassuring text like “No credit card required” next to a submission) build trust and reduce friction【42†L416-L424】【42†L421-L430】. Clear, specific calls-to-action (“Request My Dossier” rather than just “Submit”) drive engagement【42†L504-L513】. Multi-step forms (if needed) should include progress indicators to reduce anxiety【44†L13-L18】【44†L25-L28】. 

For *research intake workflows*, an effective form collects the full context of a research request. Experts advise sections for **project overview** (research question, objectives, background) and **target audience or subjects**【59†L87-L96】, plus methodology preferences, timeline/budget, and stakeholder/approval info【59†L111-L119】【59†L119-L125】. Using templates for *case briefs*, *prioritization*, and *approval flows* helps standardize the process. In other words, the intake system should gather who/what/where/when and *why*, not just free-form text. 

Finally, as a portfolio-quality vanilla frontend, the site is expected to use **semantic HTML**, modern CSS (Flexbox/Grid layouts, responsive design) and unobtrusive JavaScript. Proper tags (<header>, <nav>, <section>, etc.), alt attributes on images, and clean separation of content and style are basic requirements【63†L262-L270】. A well-crafted vanilla site will also perform well on accessibility and SEO (semantic headings and labels pay dividends【63†L262-L270】【55†L121-L124】).

# Form and workflow analysis  
The current forms (assumed from context) seem to collect free-text data on cases or persons of interest, but they’re likely not optimally structured. Based on best practices, *key categories* appear to be missing or incomplete. For example, an effective research intake form should include: **Project Overview** (what is the core question or problem, objectives, background context)【59†L87-L96】; **Subject/Target Details** (demographics or roles of persons/organizations involved)【59†L95-L103】; **Method and Resources** (type of research, tools to use, any constraints)【59†L103-L111】; **Timeline/Priority** (deadlines, milestones, urgency)【59†L111-L119】; and **Stakeholders** (who is submitting this request, who must approve, any ethical review needed)【59†L119-L125】. 

If the site’s forms currently only ask for a free-form description and maybe an author name, then vital structure is missing. For a dossier workflow, it’s important to break fields into logical groups. For instance, grouping all identity data (names, aliases, affiliations) together and all event data (date, location, evidence links) together makes later processing easier【52†L1-L4】【53†L1-L4】. Sequencing also matters: NNG advises keeping forms short and “remove fields which collect information that can be derived or omitted”【51†L1-L4】. If our form is too long or buries simple fields among many, it discourages completion. Likewise, questions should follow a natural research order – e.g. ask “What is the scenario?” before “Who is involved?” rather than vice versa. 

**Missing information:** Downstream analysis would benefit from data like: *case title*, *summary of issue*, *relevant dates/locations*, *classification (e.g. privacy/sensitivity level)*, and *supporting links or files*. The site should allow structured inputs: use a date picker for dates, dropdowns or checkboxes for fixed categories (e.g. type of event), and multi-line textareas for narrative. If fields like “POI list” are comma-separated text, it would be better as separate inputs or tags. Current fields should be reviewed to see if they capture all recommended sections from the research-intake guide【59†L87-L96】【59†L119-L125】. 

**Workflow fit:** For integration with automated agents, the output needs consistency. That means each field should map cleanly to a data field or JSON key. For example, instead of one giant “Details” box, have discrete fields like “Name”, “Affiliation”, “Role”, “Description”, etc. that can be parsed automatically. Any optional fields should be clearly marked, and conditional logic (showing only relevant fields when needed) could prevent confusion. As-is, the current form structure seems too free-form (“open narrative”) to support a methodical analysis without substantial human cleanup. 

**Improved intake architecture (suggestions):**  
- *Section 1: Case Overview* – Fields for **Case Title**, **Brief Description** (research question/problem statement), **Objectives/Goals** (what questions need answering)【59†L87-L96】. These set the mission.  
- *Section 2: Subjects or POIs* – For each person/organization: **Name**, **Aliases**, **Known Affiliations/Group**, **Role in the case**, and **Background**. Allow adding multiple entries if needed.  
- *Section 3: Details & Context* – **Event or Location** (including date/time inputs), **Category** (select from predefined types like “Political”, “Social Media”, etc.), and **Evidence/References** (URLs or file descriptions).  
- *Section 4: Methodology & Priority* – A dropdown or checkboxes for **Analysis Type** (e.g. open-source intel, interview, etc.), **Deadline/Urgency** (date picker or priority level), and **Available Resources**.  
- *Section 5: Stakeholders & Contact* – **Submitted by** (name/email), **Organizational Unit**, and any **Required Approvals** (e.g. “yes/no” toggle for ethics review). This helps track responsibility.  
- *Section 6: Additional Notes* – A catch-all **Comments** box and *Data sensitivity* (non-sensitive, confidential, etc.).  

Use appropriate input types: e.g. `type="date"`, `<select>` for choices, text areas for narratives. Group related fields visually (fieldsets or bordered sections)【52†L1-L4】, and enforce single-column alignment【53†L1-L4】. Where possible, constrain inputs (e.g. character limits, pattern validation) so future agents can reliably parse them. Mark very short/optional fields as such to avoid overburdening users. In summary, the form should mirror the structured approach in the research-intake best practices【59†L87-L96】【59†L119-L125】, rather than a flat laundry list of questions.

The trust/capacity gap is huge: as Kask notes, amateur sites make visitors question competence【36†L37-L46】. In short, there are multiple “trust breakers”: amateur visuals, generic wording (which recalls AI content)【38†L177-L184】, no credential-building content, and no indication of legitimacy or oversight. 

Overall, the tone and style risk being misread. Without explicit cues, some users might think it’s a hobby “fan site” rather than a serious tool. The visual style (if very minimal or stark) clashes with the idea of a “mission-driven” project, creating confusion about the site’s identity.
